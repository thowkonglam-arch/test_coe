🎯 North Star — Test Scenario & Case Generation (V-Model Aligned)
The COE Test Approach governs how scenarios are derived. Agentic AI ingests the Project Light BRD, user stories, and integration specs from the V-model's left branch (Requirements → Functional Design → Technical Design) and maps each requirement to test scenarios at every level: component (UT), integration (SIT), and acceptance (UAT).

For Project Light: Consumer journey scenarios — app login, DSP coupon redemption, PayNow checkout, loyalty earn, UWMS fulfilment — are generated systematically from business requirements, with normal, boundary, and exception flows covered. COE reviews and approves. Output: a version-controlled test case library in ADO/Jira+Xray, ready before SIT opens.

North Star outcome: Coverage gaps caught at requirement stage — not discovered in UAT. Golden scenario library established from day one, traceable to BRD.
✕ close
🎯 North Star — UT/ST Automation (V-Model Left Branch)
AI generates unit and string test stubs directly from component-level technical specs (V-model left branch). For each API endpoint, service method, or UI component defined in the technical design, the AI produces executable test scaffolding — reducing manual coding effort for the vendor teams.

For Project Light: DSP loyalty API tests, OneSAP price feed validators, UWMS trigger verification, and payment tokenisation checks are auto-scaffolded by AI. COE reviews, approves, and commits to the test repo. Execution is CI/CD-integrated — results feed the COE dashboard automatically.

North Star outcome: Developer-tested components before SIT entry. Integration risks surface at string test level — not during SIT. Consistent exit evidence for every story.
✕ close
🎯 North Star — Golden Set Regression Automation
The COE's Golden Scenario Library is the regression baseline. AI selects the relevant golden set subset for each release (based on delta analysis), generates optimised regression scripts, and maintains them as the app evolves. AI also identifies new scenarios to add based on defect history and V-model coverage gaps.

For Project Light: Every sprint regression runs the applicable golden set: checkout E2E, loyalty earn/redeem, PayNow + GrabPay payment matrix, DSP promo application, UWMS fulfilment trigger. AI maintains these scripts automatically — no manual upkeep as the app evolves. Pre-Prod gate requires full golden set pass before COE endorses go-live.

North Star outcome: Zero manual regression triage. Defect escapes trend down every release. Go-live endorsed on evidence, not vendor confidence.
✕ close
🎯 North Star — Agentic AI for Defect Management
Agentic AI transforms defect management from reactive triage to predictive quality intelligence. AI continuously ingests the live defect stream across all Project Light sprint cycles — analysing patterns, velocity, severity distributions, and component hotspots to surface actionable insights in real time.

Key AI capabilities:
• Automated trend analysis: AI detects defect density spikes by component (DSP promo engine, PayNow gateway, UWMS fulfilment trigger) and raises COE alerts before manual review catches them — reducing triage lag from days to hours.
• Severity drift detection: AI flags when vendors systematically downgrade Sev-1s (consumer checkout failures) and prompts COE escalation with supporting evidence.
• Escape rate prediction: Using defect escape patterns from prior sprints, AI predicts the likelihood of SIT-to-UAT escapes in the current cycle — enabling targeted hardening before UAT opens.
• Root cause clustering: AI groups similar defects across DSP, UWMS, and OneSAP integration points to surface cross-system root causes that siloed vendor teams miss.
• Remediation recommendations: AI suggests entry criteria tightening for the next sprint based on historical escape data and V-model coverage gaps.

North Star outcome: Programme leadership sees a predictive quality signal — not just a defect count. Defect escapes from SIT to UAT trend down every release. Triage time halved.

Pillar 3 — Test Types & Coverage

Test Coverage & Risk

Every test type. Every risk. Before production.

⭐ Tangible North Star Outcome — What Will Exist in Real Life

Each release is assessed against an agreed test scope. Functional and integration testing are standard. Non-functional testing (security, resilience, load) is applied where risk warrants it. Scope is agreed with programme leadership ahead of each release cycle.

📋
Functional
SIT — system-level verification covering each domain (Finance, Retail, Store Ops, Supply Chain) and cross-domain integration paths.

UAT — business-led acceptance confirming end-to-end process flows meet stakeholder expectations. COE governs entry/exit; Business Owner signs off.

Regression — structured re-test of established scenarios after any change. Scope determined by COE risk assessment — wider for major changes, targeted for hotfixes.
⚡
Non-Functional
Performance & Load — validates critical systems (iPOS, DSP, OneSAP, UWMS) handle expected throughput at peak periods (CNY, EOFY). Scope agreed per release with programme leadership.

Security — OWASP-aligned testing and vulnerability assessment for externally-facing and payment systems. Findings resolved before go-live per CSA Minimum Cybersecurity Standards.

Disaster Recovery — failover and recovery procedures exercised. RTO targets agreed with business — stores must have a tested fallback before Wave rollout.
🔗
Integration
End-to-End Cross-System — COE-governed scenarios traversing multiple systems (checkout: Light → DSP → UWMS → OneSAP). Part of the golden set — every release assessed against applicable E2E paths.

Interface Resilience — DSP as the central orchestration layer is a shared dependency. Message integrity, retry behaviour, and failure modes validated across all connected systems.

🔥 Resiliency Testing — A Release Gate, Not a Nice-to-Have

What happens to 160+ FPG stores if GCP goes down during CNY? If the answer isn't "we tested it" — it's a risk. Resiliency scope is proportional to release risk, agreed per cycle with programme leadership.

🛡 Disaster Recovery
Failover procedures tested under controlled conditions. RTO targets agreed with business. Evidence reviewed by COE. Stores must have a validated fallback before each Wave deployment.
🔲 Offline Mode (iPOS)
Store continues transacting when network drops. Local transaction queue captures sales and syncs on restore — no data loss. Tested before every applicable iPOS release.
⚡ Peak Load
CNY, EOFY and campaign peaks profiled with FPG. Load tests run against critical paths (iPOS checkout, DSP loyalty, UWMS fulfilment) at projected peak throughput — capacity gaps surfaced before production.
🔒 Security Validation
For externally-facing systems (Light, iPOS) and payment flows: vulnerability assessment and penetration testing. Critical and high findings resolved before go-live gate. Aligned to CSA Minimum Cybersecurity Standards.
🔗 Integration Resilience
DSP is a shared integration layer across all FPG initiatives. Circuit breaker and retry behaviour validated. Interface failure modes — API timeout, payload rejection, dead-letter handling — tested before production.
📋 Release-Risk Scoping
Not all resiliency tests run for every release. COE assesses change risk per cycle and defines the applicable scope. Major feature releases trigger full scope; config changes run targeted subsets only.

Golden scenarios are defined early from requirements and system design — not written after testing starts. This is the V-model discipline: test scenarios trace back to requirements before a line of code is built.

🏆 What makes a scenario "golden"?

1
Critical business outcome — failure impacts customers or revenue
2
Crosses system boundaries — not just a unit test
3
Business-approved — agreed with Business Owner, not just project team
4
Version-controlled — any change requires COE approval
5
Traced to a requirement — defined at system design phase, not post-build

🔨 Change-type trigger rule

FeatureFull golden set for the affected application
ConfigGolden subset for impacted flows only

⭐ The Pre-Production Gate — FPG's Final Shakeout Before Every Go-Live

🗄 COE Entry Checklist
  • All 5 systems at production-equivalent versions — confirmed by COE version matrix
  • Production-derived data loaded and validated (per governance policy)
  • No open P1/P2 defects carried in from UAT — gate is hard, not advisory
📋 COE Exit Clearance Criteria
  • Full golden scenario set executed and passed — no critical flow exceptions
  • Performance spot-check against agreed peak-load thresholds
  • COE issues written Pre-Prod clearance — prerequisite for go-live endorsement

📅 Version Matrix & Freeze Controls

  • Live version matrix: COE publishes and maintains a cross-initiative environment version register, updated fortnightly. Any delta between deployed version and integration-gate target is surfaced as a blocker — not discovered during SIT execution
  • 48-hour freeze window: COE enforces configuration and deployment freeze ahead of every SIT and Pre-Prod cycle. Violations require COE exception approval — escalation path is to ITMO
  • Environment provisioning accountability: Infrastructure sizing, cloud provisioning, and inter-system version alignment remain the accountability of FPG D&T and SI vendors. COE governs the standards and enforces the gate — not environment operations

🔗 FPG's 5-System Integration Complexity

DSP, OneSAP, UWMS, Light and iPOS share tightly coupled integration layers. A version mismatch on any one system can silently corrupt an entire SIT cycle. The COE's version matrix makes this risk visible before testing begins — not after a failed SIT run wastes a two-week cycle.

Pillar 7 — Test Data Management

Test Data Management

COE defines the data requirements and holds the gate — provisioning accountability sits with FPG D&T and SI vendors.

⭐ Tangible North Star Outcome — What Will Exist in Real Life

The COE defines and enforces a test data strategy anchored in production-derived datasets, where operationally feasible and approved under FPG's data governance policy. The COE does not provision data — it specifies data requirements, validates readiness before SIT gates open, and coordinates with FPG data owners and SI vendors to resolve gaps. No SIT cycle commences without COE confirmation that minimum data readiness criteria are met. Cross-system referential integrity across DSP, OneSAP, UWMS, LIGHT, and iPOS is verified by automated checks run as a pre-gate control — not discovered during execution.

In large retail transformations, test data gaps are among the most common causes of SIT delays. For FPG, the risk is compounded: OneSAP R1 and R2 migration waves, DSP loyalty platform migration, and UWMS cut-over all introduce migrated datasets that must be validated in lower environments before they reach UAT or Pre-Prod — where a data failure costs far more to fix.

📦 COE Data Strategy — Three Anchors

① Production-derived where feasible
COE specifies which datasets must be production-derived (SKU catalogue, price tables, promo config, loyalty balances) — and coordinates access approvals with FPG data owners before SIT planning, not as a last-minute blocker
② Migration wave validation — each wave, every time
OneSAP R1/R2, DSP, and UWMS migration outputs are validated by COE-coordinated integrity checks in lower environments. Structural defects are caught before they derail UAT — where fixing them is 10× more expensive than in SIT
③ Agreed synthetic alternative — risk-documented
Where production-derived data is not available, the COE defines an agreed synthetic dataset as a fallback — but documents this explicitly as a risk to go-live confidence, escalated to ITMO
