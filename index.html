📋 8 Quality Pillars — What a Great COE Delivers for Project Light Click to expand ▼

Each pillar traces from the COE Charter to a tangible outcome — chartered governance, standardised gates, golden scenarios covering all consumer journeys, and version-aligned environments across DSP, UWMS, and OneSAP. Click any pillar to expand.

P1🏛Governance — COE Charter from sprint 1; no go-live on vendor self-certification▼

Pillar output: A signed COE Charter establishes clear quality ownership from sprint 1 — COE sets standards, FPG D&T owns system sign-off, Business Owner owns UAT acceptance. No go-live is vendor self-certified.

  • Sprint-level gates: COE Test Manager reviews evidence every sprint; stories not evidenced do not advance
  • Weekly QA Council: integration risks with DSP, UWMS, OneSAP surfaced before they block SIT
  • Go-live gate: COE quality endorsement + FPG D&T + Business Owner UAT acceptance — all three required
P2📐Standards — One consistent test standard across all vendors; phase gates cannot be skipped▼

Pillar output: Entry and exit criteria defined once by COE and applied consistently. P1 = consumer-impacting (checkout failure, loyalty earn break, payment error). Vendors cannot self-downgrade severity.

  • SIT exit: zero P1, integration endpoints confirmed, critical scenarios passed, evidence archived
  • UAT exit: business sign-off, full payment matrix and loyalty journey passed
  • No gate-skipping under delivery pressure — exceptions require COE Lead and programme leadership acknowledgement
P3🔍Coverage — Consumer journeys, API integration, performance and security — scoped per release risk▼

Pillar output: COE-defined coverage matrix — the right test types for a B2C app handling payments, loyalty, and personal data. Scope agreed per release with programme leadership; not delegated to the vendor.

  • Functional: checkout, loyalty earn/redeem, all payment types — tested on iOS, Android, mobile web each release
  • API & integration: DSP loyalty APIs, UWMS fulfilment trigger, OneSAP price feed validated before SIT
  • Performance: CNY and campaign launch peaks agreed with FPG and validated before go-live
  • Security: OWASP Top 10, payment tokenisation, PCI DSS evidence compiled before go-live gate
P4⚙️Tooling — Standardised toolset and unified dashboard; quality visible without chasing vendors▼

Pillar output: One test management workspace for all cases, results, and defects — linked to user stories. A unified COE quality dashboard gives FPG D&T and programme leadership a live view without manual status chasing.

  • ADO / Jira+Xray: COE reviews evidence in the same system vendors use — no translation layer
  • UI automation: regression coverage built progressively for core journeys (checkout, loyalty, payment) on iOS, Android, web
  • Dashboard: RAG per sprint/release, defect severity, execution progress, environment status — published to programme Health Check
P5🏆Golden Scenarios — AI-augmented library; checkout, loyalty, payments, and E2E cross-system — must pass before any release▼

Pillar output: A version-controlled COE-curated golden scenario library — V-model aligned, AI-augmented. Any release is expected to pass the applicable subset before production. Exceptions require COE and programme leadership acknowledgement.

  • Registration, login (OTP / biometric), session management
  • Checkout end-to-end: cart → address → payment (PayNow, GrabPay, NETS, FairPrice wallet) → order confirmation
  • Loyalty earn: purchase → DSP credit → balance updated in-app; Loyalty redeem: points at checkout → DSP deduction → order total correct
  • Integration E2E: checkout including DSP loyalty earn, OneSAP price validation, UWMS fulfilment trigger
P6🌏Environment Alignment — Version-aligned across Light, DSP, UWMS, OneSAP; no mid-cycle conflicts▼

Pillar output: COE-defined environment alignment strategy — versions of Light, DSP, UWMS, and OneSAP are aligned across DEV, SIT, UAT, and Pre-Prod before each window opens. No deployment overwrites a running SIT cycle.

  • Environment matrix published to all teams before any SIT window opens — 48-hr freeze before entry
  • UAT: no stubs — real DSP, UWMS, OneSAP endpoints; business tests actual promo and loyalty behaviour
  • Pre-Prod: production-equivalent versions across all systems; full golden set + performance spot-check; COE endorses before go-live
P7🗃️Test Data — Migration-validated loyalty and product data ready before SIT; real consumer behaviour tested▼

Pillar output: Production-derived datasets — including migrated loyalty and product data — available and integrity-checked before SIT begins. Real consumer behaviour tested; a corrupted loyalty balance is caught in testing, not by a customer.

  • Migrated loyalty data validated in lower environments before UAT — balance integrity confirmed
  • Active DSP promo configurations and OneSAP price tables aligned to SIT version — real checkout behaviour, not generic test codes
  • Edge cases pre-seeded: zero and max loyalty balances, redemption threshold boundary, multi-payment profiles, blocked accounts
P8📊Metrics — Objective dashboard; go-live decided on evidence; sprint-to-production escape rate tracked▼

Pillar output: Live quality dashboard — daily for test team, weekly for programme leadership. Go-live gate requires: zero P1, UAT business sign-off, Pre-Prod golden set passed, security evidence compiled — not a vendor "green" status.

  • Daily: open defects by severity, execution progress, blocked cases, environment status
  • Weekly: RAG per sprint/release, go-live readiness, automation coverage — feeds programme Health Check
  • Defect escape tracking: SIT → UAT → production escape rate tracked per release; upward trend triggers tighter entry criteria

📋 8 Quality Pillars — What a Great COE Delivers for iPOS (Project Turbo) Click to expand ▼

Each pillar traces from the COE Charter to a tangible outcome for iPOS — chartered governance, standardised gates, golden scenarios, environment alignment, and objective quality evidence. Click any pillar to expand.

P1🏛Governance — COE Charter establishes quality ownership; no deployment approved on verbal confidence▼

Pillar output: The COE Charter defines who governs quality across all 9 initiatives. For iPOS, every deployment requires COE quality endorsement — no deployment goes live on vendor verbal sign-off.

  • COE Test Manager reviews all readiness evidence before any deployment proceeds
  • Go-live gate: zero P1, regression closed, DR tested, security items cleared — evidence-based
  • BAU change quality: any post-deployment update to iPOS goes through COE impact assessment
P2📐Standards — Consistent defect taxonomy and phase gates applied uniformly across all vendors▼

Pillar output: One standardised test strategy. P1 is defined once and applies to every vendor — a store-impacting defect cannot be reclassified as P2 without COE approval. Every deployment follows the same exit gate.

  • SIT exit: zero P1, all transaction types passed, integration scenarios closed, evidence archived
  • UAT exit: business sign-off, full payment matrix passed, offline-mode confirmed
  • Defect SLA uniform across all iPOS vendors — P1 escalated within 4 hours regardless of workstream
P3🔍Coverage — All payment types, offline mode, DR, and integration — right scope per release risk▼

Pillar output: COE defines coverage scope per release. Full golden set for major changes; targeted subset for hotfixes. DR and offline-mode are release gates — not optional at 160-store scale.

  • Functional: Cash, Card, NETS, PayNow QR, GrabPay, multi-tender, void, refund, Z-report, loyalty earn/redeem
  • Offline mode: transact offline, queue syncs on restore — no data loss. Tested before every applicable deployment
  • DR / GCP failover: stores must have a tested fallback. RTO targets agreed and evidenced before deployment
  • Performance: 160-store peak load profiled and validated before mass rollout
P4⚙️Tooling — Standardised toolset across all vendors; one live dashboard for programme leadership▼

Pillar output: Every change has a linked test plan in the standardised toolset. Programme leadership sees readiness, defect counts, and regression status without manual consolidation from vendor teams.

  • ADO / Jira+Xray: single workspace; COE reviews evidence where vendors work — no translation layer
  • POS-compatible automation: regression cycles accelerated for golden iPOS scenarios progressively
  • Live dashboard: open P1/P2, regression pass rate, environment availability, readiness RAG — visible to programme leadership
P5🏆Golden Scenarios — Version-controlled library covering every payment type, offline, DR, and integration path▼

Pillar output: A COE-curated, version-controlled iPOS golden scenario library — V-model aligned, AI-augmented. Any change is assessed against its applicable golden subset before deployment. No deployment skips this gate.

  • Core: Cash · Card/NETS · PayNow QR · GrabPay · multi-tender · void · refund · loyalty earn/redeem
  • Offline mode & sync: store transacts without network, queue captured locally, syncs on reconnection
  • Integration: DSP promo applied at checkout, OneSAP price validated, Release 4 coupon engine tested before deployment
  • AI-augmented generation accelerates library build; COE reviews and approves all scenarios before use
P6🌏Environment Alignment — Version matrix across Release 2/3/4; no contamination at SIT gates▼

Pillar output: COE-defined environment governance policy. iPOS Release 2, 3, and 4 are version-aligned; no deployment overwrites a running SIT cycle. Pre-Prod is the final shakeout gate before every deployment.

  • Environment matrix published to all teams before any SIT window opens — version per system per environment
  • 48-hr freeze window before UAT and Pre-Prod — no mid-cycle deployments
  • Pre-Prod: production-equivalent versions of iPOS, DSP, OneSAP — full golden set executed; COE endorses before deployment
P7🗃️Test Data — Production-derived SKU, pricing, and promo data; real store behaviour tested▼

Pillar output: Production-derived iPOS test data — real SKU catalogue, pricing tables, DSP promo configs — available before SIT opens. Real store checkout behaviour tested, not synthetic data that always passes.

  • Active DSP promo configurations loaded in SIT — discounts and coupon logic validated against real promo data
  • OneSAP price tables aligned to SIT version — price changes tested before they reach stores
  • Edge cases pre-seeded: zero and max loyalty balances, multiple-tender splits, blocked accounts, return-on-card
P8📊Metrics — Readiness decided on objective evidence; quality trends visible to programme leadership▼

Pillar output: Live quality dashboard — daily view for test team, weekly for programme leadership. Deployment readiness is an evidence package: P1 count, golden set result, DR test status, security items — not a verbal assurance.

  • Daily: open defects by severity, execution progress, blocked scenarios, environment status
  • Weekly: readiness RAG, regression completion, automation coverage — feeds programme Health Check
  • Post-deployment: production defect rate tracked; escapes feed into next release's golden set refinement

Pillar 3 — Test Types & Coverage

Test Coverage & Risk

Every test type. Every risk. Before production.

⭐ Tangible NorthStar Outcome — What Will Exist in Real Life

Each release is assessed against an agreed test scope. Functional and integration testing are standard. Non-functional testing (security, resilience, load) is applied where risk warrants it. Scope is agreed with programme leadership ahead of each release cycle.

📋
Functional
SIT — system-level verification covering each domain (Finance, Retail, Store Ops, Supply Chain) and cross-domain integration paths.

UAT — business-led acceptance confirming end-to-end process flows meet stakeholder expectations. COE governs entry/exit; Business Owner signs off.

Regression — structured re-test of established scenarios after any change. Scope determined by COE risk assessment — wider for major changes, targeted for hotfixes.
⚡
Non-Functional
Performance & Load — validates critical systems (iPOS, DSP, OneSAP, UWMS) handle expected throughput at peak periods (CNY, EOFY). Scope agreed per release with programme leadership.

Security — OWASP-aligned testing and vulnerability assessment for externally-facing and payment systems. Findings resolved before go-live per CSA Minimum Cybersecurity Standards.

Disaster Recovery — failover and recovery procedures exercised. RTO targets agreed with business — stores must have a tested fallback before Wave rollout.
🔗
Integration
End-to-End Cross-System — COE-governed scenarios traversing multiple systems (checkout: Light → DSP → UWMS → OneSAP). Part of the golden set — every release assessed against applicable E2E paths.

Interface Resilience — DSP as the central orchestration layer is a shared dependency. Message integrity, retry behaviour, and failure modes validated across all connected systems.

🔥 Resiliency Testing — A Release Gate, Not a Nice-to-Have

What happens to 160+ FPG stores if GCP goes down during CNY? If the answer isn't "we tested it" — it's a risk. Resiliency scope is proportional to release risk, agreed per cycle with programme leadership.

🛡 Disaster Recovery
Failover procedures tested under controlled conditions. RTO targets agreed with business. Evidence reviewed by COE. Stores must have a validated fallback before each Wave deployment.
🔲 Offline Mode (iPOS)
Store continues transacting when network drops. Local transaction queue captures sales and syncs on restore — no data loss. Tested before every applicable iPOS release.
⚡ Peak Load
CNY, EOFY and campaign peaks profiled with FPG. Load tests run against critical paths (iPOS checkout, DSP loyalty, UWMS fulfilment) at projected peak throughput — capacity gaps surfaced before production.
🔒 Security Validation
For externally-facing systems (Light, iPOS) and payment flows: vulnerability assessment and penetration testing. Critical and high findings resolved before go-live gate. Aligned to CSA Minimum Cybersecurity Standards.
🔗 Integration Resilience
DSP is a shared integration layer across all FPG initiatives. Circuit breaker and retry behaviour validated. Interface failure modes — API timeout, payload rejection, dead-letter handling — tested before production.
📋 Release-Risk Scoping
Not all resiliency tests run for every release. COE assesses change risk per cycle and defines the applicable scope. Major feature releases trigger full scope; config changes run targeted subsets only.

Golden scenarios are defined early from requirements and system design — not written after testing starts. This is the V-model discipline: test scenarios trace back to requirements before a line of code is built.

🏆 What makes a scenario "golden"?

4

🤖 How Gen AI Helps Build & Maintain the Golden Set

Scenario Definition from Requirements
Gen AI analyses user stories, API specs, and process flows to draft candidate golden scenarios. COE reviews, selects, and approves — human judgement on criticality remains essential.
Test Case Generation
From approved golden scenarios, Gen AI generates detailed test cases (steps, data inputs, expected results) — accelerating the build of the scenario library particularly for new systems like Light, DSP, and UWMS.
Automation Script Generation
AI-assisted code generation for regression automation scripts targeting golden scenarios. Reduces the time to build an automation-supported regression suite. COE reviews and validates all generated scripts before use.
Library Maintenance
When system designs change (e.g. new DSP API version, OneSAP Retail R2 go-live), AI-assisted impact analysis flags which golden scenarios may need updating — preventing stale test libraries.

Test Data Management

Real data behaviour tested before it hits production.

Production-derived datasets — including migrated data — available and validated before SIT begins. Test data reflects real system behaviour across DSP, OneSAP, UWMS, Light and iPOS. No SIT cycle delayed because data was missing, structurally wrong, or not yet migrated.
